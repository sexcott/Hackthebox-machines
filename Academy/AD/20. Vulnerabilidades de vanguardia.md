Cuando se trata de ciclos y administración de parches, muchas organizaciones no se apresuran a implementar parches a través de sus redes. Debido a esto, es posible que podamos lograr una ganancia rápida ya sea para el acceso inicial o para la escalada de privilegios de dominio utilizando una táctica muy reciente. En el momento de redactar este artículo (abril de 2022), las tres técnicas que se muestran en esta sección son relativamente recientes (dentro de los últimos 6 a 9 meses). Estos son temas avanzados que no se pueden cubrir a fondo en una sección de módulo. El propósito de demostrar estos ataques es permitir a los estudiantes probar los últimos y mejores ataques en un entorno de laboratorio controlado y presentar temas que se cubrirán con extrema profundidad en módulos más avanzados de Active Directory. Al igual que con cualquier ataque, si no comprende cómo funcionan o el riesgo que podrían representar para un entorno de producción, sería mejor no intentarlos durante una interacción con un cliente en el mundo real. Dicho esto, estas técnicas podrían considerarse "seguras" y menos destructivo que ataques como [Zerologon](https://www.crowdstrike.com/blog/cve-2020-1472-zerologon-security-advisory/) o [DCShadow](https://stealthbits.com/blog/what-is-a-dcshadow-attack-and-how-to-defend-against-it/). Aun así, siempre debemos tener precaución, tomar notas detalladas y comunicarnos con nuestros clientes. Todos los ataques conllevan un riesgo. Por ejemplo, el ataque `PrintNightmare` podría bloquear el servicio de cola de impresión en un host remoto y provocar una interrupción del servicio.

Como profesionales de la seguridad de la información en un campo que cambia y evoluciona rápidamente, debemos mantenernos alerta y al tanto de los ataques recientes y de las nuevas herramientas y técnicas. Recomendamos probar todas las técnicas de esta sección y realizar investigaciones adicionales para encontrar otros métodos para realizar estos ataques. Ahora, profundicemos.

## NoPac (suplantación de nombre de cuenta Sam)

Un gran ejemplo de una amenaza emergente es la [vulnerabilidad Sam_The_Admin](https://techcommunity.microsoft.com/t5/security-compliance-and-identity/sam-name-impersonation/ba-p/3042699), también llamada `noPac` o conocida como `SamAccountName Spoofing` lanzada a finales de 2021. Esta vulnerabilidad abarca dos CVE [2021-42278](https://msrc.microsoft.com/update-guide/vulnerability/CVE-2021-42278) y [2021-42287](https://msrc.microsoft.com/update-guide/vulnerability/CVE-2021-42287), lo que permite la escalada de privilegios dentro del dominio desde cualquier usuario de dominio estándar al acceso de nivel de administrador de dominio con un solo comando. A continuación se muestra un desglose rápido de lo que proporciona cada CVE con respecto a esta vulnerabilidad.

|42278|42287|
|---|---|
|`42278`es una vulnerabilidad de omisión con el Administrador de cuentas de seguridad (SAM).|`42287`es una vulnerabilidad dentro del Certificado de atributo de privilegio (PAC) de Kerberos en ADDS.|

Esta ruta de explotación aprovecha la posibilidad de cambiar la `SamAccountName` de una cuenta de computadora a la de un controlador de dominio. De forma predeterminada, los usuarios autenticados pueden agregar hasta [diez computadoras a un dominio](https://docs.microsoft.com/en-us/windows/security/threat-protection/security-policy-settings/add-workstations-to-domain). Al hacerlo, cambiamos el nombre del nuevo host para que coincida con el SamAccountName del controlador de dominio. Una vez hecho esto, debemos solicitar tickets de Kerberos provocando que el servicio nos emita tickets con el nombre del DC en lugar del nuevo nombre. Cuando se solicita un TGS, emitirá el billete con el nombre más cercano. Una vez hecho esto, tendremos acceso como ese servicio e incluso podremos recibir un shell de SISTEMA en un controlador de dominio. El flujo del ataque se describe en detalle en esta [entrada de blog](https://www.secureworks.com/blog/nopac-a-tale-of-two-vulnerabilities-that-could-end-in-ransomware).

Podemos utilizar esta [herramienta](https://github.com/Ridter/noPac) para realizar este ataque. Esta herramienta está presente en el host ATTACK01 en `/opt/noPac`.

Una vez que Impacket esté instalado y nos aseguremos de que el repositorio esté clonado en nuestro cuadro de ataque, podemos usar los scripts en el directorio NoPac para verificar si el sistema es vulnerable usando un escáner (`scanner.py` ) luego usa el exploit (`noPac.py`) para obtener un shell como `NT AUTHORITY/SYSTEM`. Podemos utilizar el escáner con una cuenta de usuario de dominio estándar para intentar obtener un TGT del controlador de dominio de destino. Si tiene éxito, esto indica que el sistema es, de hecho, vulnerable. También notaremos que el número `ms-DS-MachineAccountQuota` está establecido en 10. En algunos entornos, un administrador de sistemas astuto puede establecer el valor `ms-DS-MachineAccountQuota` en 0. Si este es el En este caso, el ataque fallará porque nuestro usuario no tendrá los derechos para agregar una nueva cuenta de máquina. Establecer esto en `0` puede evitar bastantes ataques de AD.

## Windows Defender y Consideraciones sobre SMBEXEC.py

Si Windows Defender (u otro producto AV o EDR) está habilitado en un objetivo, es posible que se establezca nuestra sesión de shell, pero es probable que se produzca un error al emitir cualquier comando. Lo primero que hace smbexec.py es crear un servicio llamado `BTOBTO`. Se crea otro servicio llamado `BTOBO` y cualquier comando que escribamos se envía al destino a través de SMB dentro de un archivo .bat llamado `execute.bat`. Con cada nuevo comando que escribimos, se crea un nuevo script por lotes y se repite en un archivo temporal que ejecuta dicho script y lo elimina del sistema. Miremos un registro de Windows Defender para ver qué comportamiento se consideró malicioso.

## PetitPotam (MS-EFSRPC)
PetitPotam ([CVE-2021-36942](https://msrc.microsoft.com/update-guide/vulnerability/CVE-2021-36942)) es una vulnerabilidad de suplantación de LSA que se parchó en agosto de 2021. La falla permite que un atacante no autenticado obligar a un controlador de dominio a autenticarse contra otro host utilizando NTLM a través del puerto 445 a través del [Protocolo remoto de la autoridad de seguridad local (LSARPC)](https://docs.microsoft.com/en-us/openspecs/windows_protocols/ms-lsad/1b5471ef-4c33-4a91-b079-dfcbb82f05cc) abusando del de Microsoft. [Protocolo remoto de cifrado del sistema de archivos (MS-EFSRPC)](https://docs.microsoft.com/en-us/openspecs/windows_protocols/ms-efsr/08796ba8-01c8-4872-9221-1000ec2eff31). Esta técnica permite que un atacante no autenticado se apodere de un dominio de Windows donde [Servicios de certificados de Active Directory (AD CS)](https://docs.microsoft.com/en-us/learn/modules/implement-manage-active-directory-certificate-services/2-explore-fundamentals-of-pki-ad-cs) está en uso. En el ataque, una solicitud de autenticación del controlador de dominio objetivo se transmite a la página de inscripción web del host de la autoridad certificadora (CA) y realiza una solicitud de firma de certificado (CSR) para un nuevo certificado digital. Este certificado se puede utilizar con una herramienta como `Rubeus` o `gettgtpkinit.py` de [PKINITtools](https://github.com/dirkjanm/PKINITtools). para solicitar un TGT para el controlador de dominio, que luego puede usarse para lograr comprometer el dominio mediante un ataque DCSync.

[Esta](https://dirkjanm.io/ntlm-relaying-to-ad-certificate-services/) publicación de blog detalla más sobre la retransmisión de NTLM a AD CS y el ataque PetitPotam.

Repasemos el ataque. En primer lugar, debemos iniciar `ntlmrelayx.py` en una ventana de nuestro host de ataque, especificando la URL de inscripción web para el host de CA y utilizando la plantilla KerberosAuthentication o DomainController AD CS. Si no supiéramos la ubicación de la CA, podríamos usar una herramienta como [certi](https://github.com/zer1t0/certi) para intentar localizarla.
