Hay un problema conocido como "doble salto". problema que surge cuando un atacante intenta utilizar la autenticación Kerberos en dos (o más) saltos. El problema tiene que ver con cómo se otorgan los tickets de Kerberos para recursos específicos. Los tickets de Kerberos no deben verse como contraseñas. Son datos firmados del KDC que indican a qué recursos puede acceder una cuenta. Cuando realizamos la autenticación Kerberos, obtenemos un "ticket" que nos permite acceder al recurso solicitado (es decir, una sola máquina). Por el contrario, cuando usamos una contraseña para autenticarnos, ese hash NTLM se almacena en nuestra sesión y puede usarse en otro lugar sin problemas.

## Fondo

El "Doble Salto" El problema ocurre a menudo cuando se usa WinRM/Powershell ya que el mecanismo de autenticación predeterminado solo proporciona un ticket para acceder a un recurso específico. Es probable que esto cause problemas al intentar realizar un movimiento lateral o incluso acceder a archivos compartidos desde el shell remoto. En esta situación, la cuenta de usuario que se utiliza tiene derechos para realizar una acción pero se le niega el acceso. La forma más común de obtener shells es atacando una aplicación en el host de destino o utilizando credenciales y una herramienta como PSExec. En ambos escenarios, la autenticación inicial probablemente se realizó a través de SMB o LDAP, lo que significa que el NTLM Hash del usuario se almacenaría en la memoria. A veces tenemos un conjunto de credenciales y estamos restringidos a un método particular de autenticación, como WinRM, o preferimos usar WinRM por diversos motivos.

El quid de la cuestión es que cuando se utiliza WinRM para autenticarse en dos o más conexiones, la contraseña del usuario nunca se almacena en caché como parte de su inicio de sesión. Si usamos Mimikatz para mirar la sesión, veremos que todas las credenciales están en blanco. Como se indicó anteriormente, cuando utilizamos Kerberos para establecer una sesión remota, no utilizamos una contraseña para la autenticación. Cuando se utiliza la autenticación de contraseña, con PSExec, por ejemplo, ese hash NTLM se almacena en la sesión, por lo que cuando vamos a acceder a otro recurso, la máquina puede extraer el hash de la memoria y autenticarnos.

En términos más simples, en esta situación, cuando intentamos emitir un comando multiservidor, nuestras credenciales no se enviarán desde la primera máquina a la segunda.

Digamos que tenemos tres hosts: `Attack host` --> `DEV01` --> para enumerar el dominio, lo que requiere comunicación con el controlador de dominio, DC01. en DEV01. Queremos utilizar `DC01``Remote Management Users``PowerView`

![imagen](https://academy.hackthebox.com/storage/modules/143/double_hop.png)
Cuando nos conectamos a `DEV01` usando una herramienta como `evil-winrm`, nos conectamos con autenticación de red, por lo que nuestras credenciales no se almacenan en la memoria y, por lo tanto, , no estará presente en el sistema para autenticarse en otros recursos en nombre de nuestro usuario. Cuando cargamos una herramienta como `PowerView` e intentamos consultar Active Directory, Kerberos no tiene forma de decirle al DC que nuestro usuario puede acceder a los recursos del dominio. Esto sucede porque el ticket Kerberos TGT (Ticket Granting Ticket) del usuario no se envía a la sesión remota; por lo tanto, el usuario no tiene forma de demostrar su identidad y los comandos ya no se ejecutarán en el contexto de este usuario. En otras palabras, al autenticarse en el host de destino, el ticket del servicio de concesión de tickets (TGS) del usuario se envía al servicio remoto, lo que permite la ejecución de comandos, pero el ticket TGT del usuario no se envía. Cuando el usuario intente acceder a recursos posteriores en el dominio, su TGT no estará presente en la solicitud, por lo que el servicio remoto no tendrá forma de demostrar que el intento de autenticación es válido y se nos negará el acceso al servicio remoto.

Si la delegación sin restricciones está habilitada en un servidor, es probable que no nos enfrentemos al problema del "Doble Salto". problema. En este escenario, cuando un usuario envía su ticket TGS para acceder al servidor de destino, su ticket TGT se enviará junto con la solicitud. El servidor de destino ahora tiene el ticket TGT del usuario en la memoria y puede usarlo para solicitar un ticket TGS en su nombre en el siguiente host al que intenta acceder. En otras palabras, el ticket TGS de la cuenta se almacena en caché, que tiene la capacidad de firmar TGT y otorgar acceso remoto. En términos generales, si llegas a una casilla con delegación sin restricciones, ya ganaste y no te preocupas por esto de todos modos.
## Soluciones alternativas

En [esta publicación](https://posts.slayerlabs.com/double-hop/) se tratan algunas soluciones para el problema del doble salto. Podemos utilizar un método "anidado". `Invoke-Command` para enviar credenciales (después de crear un objeto PSCredential) con cada solicitud, por lo que si intentamos autenticarnos desde nuestro host de ataque al host A y ejecutar comandos en el host B, se nos permite. Cubriremos dos métodos en esta sección: el primero es uno que podemos usar si estamos trabajando con una sesión `evil-winrm` y el segundo si tenemos acceso GUI a un host de Windows (ya sea un host de ataque en la red o un host unido a un dominio que hemos comprometido).

## Solución #2: Registrar la configuración de PSSession

Hemos visto lo que podemos hacer para superar este problema cuando utilizamos una herramienta como `evil-winrm` para conectarnos a un host a través de WinRM. ¿Qué pasa si estamos en un host unido a un dominio y podemos conectarnos de forma remota a otro usando WinRM? ¿O estamos trabajando desde un host de ataque de Windows y nos conectamos a nuestro objetivo a través de WinRM usando el [cmdlet Enter-PSSession](https://docs.microsoft.com/en-us/powershell/module/microsoft.powershell.core/enter-pssession?view=powershell-7.2)? Aquí tenemos otra opción para cambiar nuestra configuración para poder interactuar directamente con el DC u otros hosts/recursos sin tener que configurar un objeto PSCredential e incluir credenciales junto con cada comando (lo que puede no ser una opción con algunas herramientas).

Comencemos estableciendo primero una sesión WinRM en el host remoto.

```powershell-session
PS C:\htb> Enter-PSSession -ComputerName ACADEMY-AEN-DEV01.INLANEFREIGHT.LOCAL -Credential inlanefreight\backupadm
```

Si verificamos los tickets almacenados en caché usando `klist`, veremos que existe el mismo problema. Debido al problema del doble salto, solo podemos interactuar con los recursos en nuestra sesión actual pero no podemos acceder al DC directamente usando PowerView. Podemos ver que nuestro TGS actual es bueno para acceder al servicio HTTP en el destino ya que nos conectamos a través de WinRM, que utiliza solicitudes SOAP (Protocolo simple de acceso a objetos) en formato XML para comunicarse a través de HTTP, por lo que tiene sentido.

Un truco que podemos usar aquí es registrar una nueva configuración de sesión usando el cmdlet [Register-PSSessionConfiguration](https://docs.microsoft.com/en-us/powershell/module/microsoft.powershell.core/register-pssessionconfiguration?view=powershell-7.2).

```powershell-session
PS C:\htb> Register-PSSessionConfiguration -Name backupadmsess -RunAsCredential inlanefreight\backupadm

```

Una vez hecho esto, debemos reiniciar el servicio WinRM escribiendo `Restart-Service WinRM` en nuestra PSSession actual. Esto nos expulsará, por lo que iniciaremos una nueva PSSession utilizando la sesión registrada con nombre que configuramos anteriormente.

Después de iniciar la sesión, podemos ver que el problema del doble salto se ha eliminado y si escribimos `klist`, tendremos los tickets en caché necesarios para llegar al Controlador de dominio. Esto funciona porque nuestra máquina local ahora se hará pasar por la máquina remota en el contexto del usuario `backupadm` y todas las solicitudes de nuestra máquina local se enviarán directamente al controlador de dominio.ç

	Nota: No podemos usar `Register-PSSessionConfiguration` desde un shell evil-winrm porque no podremos obtener la ventana emergente de credenciales. Además, si intentamos ejecutar esto configurando primero un objeto PSCredential y luego intentando ejecutar el comando pasando credenciales como `-RunAsCredential $Cred`, obtendremos un error porque solo podemos usar `RunAs` desde una terminal PowerShell elevada. Por lo tanto, este método no funcionará a través de una sesión evil-winrm ya que requiere acceso a la GUI y una consola PowerShell adecuada. Además, en nuestras pruebas, no pudimos lograr que este método funcionara desde PowerShell en un host de ataque Parrot o Ubuntu debido a ciertas limitaciones sobre cómo funciona PowerShell en Linux con credenciales Kerberos. Este método sigue siendo muy efectivo si estamos probando desde un host de ataque de Windows y tenemos un conjunto de credenciales o comprometemos un host y podemos conectarnos a través de RDP para usarlo como un "host de salto". para montar más ataques contra hosts en el entorno.
