## Membresía de grupo relacionado con Exchange

Una instalación predeterminada de Microsoft Exchange dentro de un entorno AD (sin modelo de administración dividida) abre muchos vectores de ataque, ya que a Exchange a menudo se le otorgan privilegios considerables dentro del dominio (a través de usuarios, grupos y ACL). El grupo `Exchange Windows Permissions` no figura como grupo protegido, pero a los miembros se les otorga la capacidad de escribir una DACL en el objeto de dominio. Esto se puede aprovechar para otorgar privilegios de DCSync al usuario. Un atacante puede agregar cuentas a este grupo aprovechando una configuración incorrecta de DACL (posible) o aprovechando una cuenta comprometida que sea miembro del grupo Operadores de cuentas. Es común encontrar cuentas de usuarios e incluso computadoras como miembros de este grupo. A menudo se agregan a este grupo usuarios avanzados y personal de soporte en oficinas remotas, lo que les permite restablecer contraseñas. Este [repositorio de GitHub](https://github.com/gdedrouas/Exchange-AD-Privesc) detalla algunas técnicas para aprovechar Exchange para escalar privilegios en un entorno AD.

El grupo Exchange `Organization Management` es otro grupo extremadamente poderoso (efectivamente, los "Administradores de dominio" de Exchange) y puede acceder a los buzones de correo de todos los usuarios del dominio. No es raro que los administradores de sistemas sean miembros de este grupo. Este grupo también tiene control total de la unidad organizativa llamada `Microsoft Exchange Security Groups`, que contiene el grupo `Exchange Windows Permissions`.

Si podemos comprometer un servidor Exchange, esto a menudo conducirá a privilegios de administrador de dominio. Además, volcar las credenciales en la memoria desde un servidor Exchange producirá decenas, si no cientos, de credenciales de texto sin cifrar o hashes NTLM. Esto suele deberse a que los usuarios inician sesión en Outlook Web Access (OWA) y Exchange almacenan en caché sus credenciales en la memoria después de un inicio de sesión exitoso.
## PrivExchange
El ataque `PrivExchange` resulta de una falla en la función Exchange Server `PushSubscription`, que permite a cualquier usuario de dominio con un buzón forzar la autenticación del servidor Exchange. a cualquier host proporcionado por el cliente a través de HTTP.

El servicio Exchange se ejecuta como SISTEMA y tiene privilegios excesivos de forma predeterminada (es decir, tiene privilegios WriteDacl en el dominio anterior a la actualización acumulativa de 2019). Esta falla se puede aprovechar para transmitir a LDAP y volcar la base de datos NTDS del dominio. Si no podemos retransmitir a LDAP, podemos aprovechar esto para retransmitir y autenticar a otros hosts dentro del dominio. Este ataque lo llevará directamente al Administrador de dominio con cualquier cuenta de usuario de dominio autenticada.
## Error de impresora

El error de la impresora es una falla en el protocolo MS-RPRN (Protocolo remoto del sistema de impresión). Este protocolo define la comunicación del procesamiento de trabajos de impresión y la gestión del sistema de impresión entre un cliente y un servidor de impresión. Para aprovechar esta falla, cualquier usuario de dominio puede conectarse a la canalización con nombre del spool con el método `RpcOpenPrinter` y usar el método `RpcRemoteFindFirstPrinterChangeNotificationEx`, y forzar al servidor a autenticarse en cualquier host proporcionado por el cliente a través de SMB.

El servicio de cola de impresión se ejecuta como SISTEMA y se instala de forma predeterminada en los servidores Windows que ejecutan Desktop Experience. Este ataque se puede aprovechar para transmitir a LDAP y otorgar a su cuenta de atacante privilegios DCSync para recuperar todos los hash de contraseña de AD.

El ataque también se puede utilizar para transmitir la autenticación LDAP y otorgar privilegios de delegación restringida basada en recursos (RBCD) para la víctima a una cuenta de computadora bajo nuestro control, otorgando así al atacante privilegios para autenticarse como cualquier usuario en la computadora de la víctima. . Este ataque se puede aprovechar para comprometer un controlador de dominio en un dominio/bosque asociado, siempre que ya tenga acceso administrativo a un controlador de dominio en el primer bosque/dominio y la confianza permita la delegación TGT, que ya no es la opción predeterminada.

Podemos utilizar herramientas como el módulo `Get-SpoolStatus` de [esta](http://web.archive.org/web/20200919080216/https://github.com/cube0x0/Security-Assessment) herramienta (que se puede encontrar en la página objetivo generado) o [esta](https://github.com/NotMedic/NetNTLMtoSilverTicket) herramienta para comprobar si hay máquinas vulnerables al error de [impresora MS-PRN](https://blog.sygnia.co/demystifying-the-print-nightmare-vulnerability). Esta falla se puede utilizar para comprometer un host en otro bosque que tenga habilitada la delegación sin restricciones, como un controlador de dominio. Puede ayudarnos a atacar a través de fideicomisos de bosques una vez que hayamos comprometido un bosque.

## MS14-068

Se trataba de una falla en el protocolo Kerberos, que podía aprovecharse junto con las credenciales de usuario de dominio estándar para elevar los privilegios a Administrador de dominio. Un ticket de Kerberos contiene información sobre un usuario, incluido el nombre de la cuenta, el ID y la pertenencia al grupo en el Certificado de atributo de privilegio (PAC). El KDC firma el PAC utilizando claves secretas para validar que el PAC no ha sido manipulado después de su creación.

La vulnerabilidad permitió que el KDC aceptara como legítimo un PAC falsificado. Esto se puede aprovechar para crear un PAC falso, presentando a un usuario como miembro de los Administradores de dominio u otro grupo privilegiado. Se puede explotar con herramientas como el [Python Kerberos Exploitation Kit (PyKEK)](https://github.com/SecWiki/windows-kernel-exploits/tree/master/MS14-068/pykek) o el kit de herramientas Impacket. La única defensa contra este ataque son los parches. La máquina [Mantis](https://app.hackthebox.com/machines/98) en la plataforma Hack The Box muestra esta vulnerabilidad.

## Rastreando credenciales LDAP
Muchas aplicaciones e impresoras almacenan credenciales LDAP en su consola de administración web para conectarse al dominio. Estas consolas suelen tener contraseñas débiles o predeterminadas. A veces, estas credenciales se pueden ver en texto sin cifrar. Otras veces, la aplicación tiene una función `test connection` que podemos usar para recopilar credenciales cambiando la dirección IP LDAP a la de nuestro host de ataque y configurando un `netcat` escucha. en el puerto LDAP 389. Cuando el dispositivo intenta probar la conexión LDAP, enviará las credenciales a nuestra máquina, a menudo en texto sin cifrar. Las cuentas utilizadas para conexiones LDAP suelen tener privilegios, pero si no, esto podría servir como punto de apoyo inicial en el dominio. Otras veces, se requiere un servidor LDAP completo para llevar a cabo este ataque, como se detalla en esta [publicación](https://grimhacker.com/2018/03/09/just-a-printer/).
## Enumerar registros DNS

Podemos utilizar una herramienta como [adidnsdump](https://github.com/dirkjanm/adidnsdump) para enumerar todos los registros DNS en un dominio utilizando una cuenta de usuario de dominio válida. Esto es especialmente útil si la convención de nomenclatura para hosts que nos devuelven en nuestra enumeración usando herramientas como `BloodHound` es similar a `SRV01934.INLANEFREIGHT.LOCAL`. Si todos los servidores y estaciones de trabajo tienen un nombre no descriptivo, nos resulta difícil saber qué atacar exactamente. Si podemos acceder a las entradas DNS en AD, potencialmente podemos descubrir registros DNS interesantes que apunten a este mismo servidor, como `JENKINS.INLANEFREIGHT.LOCAL`, que podemos utilizar para planificar mejor nuestros ataques.

La herramienta funciona porque, de forma predeterminada, todos los usuarios pueden enumerar los objetos secundarios de una zona DNS en un entorno AD. De forma predeterminada, la consulta de registros DNS mediante LDAP no devuelve todos los resultados. Entonces, al usar la herramienta `adidnsdump`, podemos resolver todos los registros en la zona y potencialmente encontrar algo útil para nuestra participación. Los antecedentes y una explicación más detallada de esta herramienta y técnica se pueden encontrar en esta [publicación](https://dirkjanm.io/getting-in-the-zone-dumping-active-directory-dns-with-adidnsdump/).

## PASSWD_NOTREQD Campo

Es posible encontrar cuentas de dominio con el campo [passwd_notreqd](https://ldapwiki.com/wiki/Wiki.jsp?page=PASSWD_NOTREQD) establecido en el atributo userAccountControl. Si se establece esto, el usuario no está sujeto a la longitud de la política de contraseñas actual, lo que significa que podría tener una contraseña más corta o ninguna contraseña (si se permiten contraseñas vacías en el dominio). Una contraseña puede estar en blanco intencionalmente (a veces los administradores no quieren que los llamen fuera de horario para restablecer las contraseñas de los usuarios) o presionar accidentalmente Intro antes de ingresar una contraseña al cambiarla a través de la línea de comando. El hecho de que esta marca esté configurada en una cuenta no significa que no se haya establecido ninguna contraseña, solo que es posible que no sea necesaria. Hay muchas razones por las que este indicador puede establecerse en una cuenta de usuario, uno de los cuales es que un producto del proveedor estableció este indicador en ciertas cuentas en el momento de la instalación y nunca lo eliminó después de la instalación. Vale la pena enumerar las cuentas con este indicador configurado y probar cada una para ver si no se requiere contraseña (lo he visto un par de veces en las evaluaciones). Además, inclúyalo en el informe del cliente si el objetivo de la evaluación es ser lo más completa posible.

## Contraseñas de preferencias de política de grupo (GPP)

Cuando se crea un nuevo GPP, se crea un archivo .xml en el recurso compartido SYSVOL, que también se almacena en caché localmente en los puntos finales a los que se aplica la política de grupo. Estos archivos pueden incluir aquellos utilizados para:

- Unidades de mapa (drives.xml)
- Crear usuarios locales
- Crear archivos de configuración de impresora (printers.xml)
- Creación y actualización de servicios (services.xml)
- Crear tareas programadas (scheduledtasks.xml)
- Cambiar las contraseñas de administrador local.

Estos archivos pueden contener una serie de datos de configuración y contraseñas definidas. El valor del atributo `cpassword` está cifrado con AES de 256 bits, pero Microsoft [publicó la clave privada AES en MSDN](https://docs.microsoft.com/en-us/openspecs/windows_protocols/ms-gppref/2c15cbf0-f086-4c74-8b70-1f2fa45dd4be?redirectedfrom=MSDN), que puede ser utilizado para descifrar la contraseña. Cualquier usuario de dominio puede leer estos archivos tal como están almacenados en el recurso compartido SYSVOL y, de forma predeterminada, todos los usuarios autenticados en un dominio tienen acceso de lectura a este recurso compartido de controlador de dominio.

Esto fue parcheado en 2014 [MS14-025 La vulnerabilidad en GPP podría permitir la elevación de privilegios](https://support.microsoft.com/en-us/topic/ms14-025-vulnerability-in-group-policy-preferences-could-allow-elevation-of-privilege-may-13-2014-60734e15-af79-26ca-ea53-8cd617073c30), para evitar que los administradores establezcan contraseñas usando GPP. El parche no elimina los archivos Groups.xml existentes con contraseñas de SYSVOL. Si elimina la política GPP en lugar de desvincularla de la unidad organizativa, la copia almacenada en caché permanece en la computadora local.

La mala higiene de las contraseñas es común en muchas organizaciones, por lo que cada vez que obtenemos credenciales, debemos verificar si podemos usarlas para acceder a otros hosts (como dominio o usuario local), aprovechar algún derecho, como ACL interesantes, recursos compartidos o use la contraseña en un ataque de pulverización de contraseñas para descubrir la reutilización de la contraseña y tal vez una cuenta que nos otorgue mayor acceso hacia nuestro objetivo.

## ASREPRoasting
Es posible obtener el Ticket de concesión de tickets (TGT) para cualquier cuenta que tenga [Do not require Kerberos pre-authentication](https://www.tenable.com/blog/how-to-stop-the-kerberos-pre-authentication-attack-in-active-directory) habilitada. Muchas guías de instalación de proveedores especifican que su cuenta de servicio se configure de esta manera. La respuesta del servicio de autenticación (AS_REP) está cifrada con la contraseña de la cuenta y cualquier usuario del dominio puede solicitarla.

Con la autenticación previa, un usuario ingresa su contraseña, que cifra una marca de tiempo. El controlador de dominio lo descifrará para validar que se utilizó la contraseña correcta. Si tiene éxito, se emitirá un TGT al usuario para futuras solicitudes de autenticación en el dominio. Si una cuenta tiene la autenticación previa deshabilitada, un atacante puede solicitar datos de autenticación para la cuenta afectada y recuperar un TGT cifrado del controlador de dominio. Esto puede estar sujeto a un ataque de contraseña fuera de línea utilizando una herramienta como Hashcat o John the Ripper.

#### Ver una cuenta con la opción No requerir autenticación previa de Kerberos

![imagen](https://academy.hackthebox.com/storage/modules/143/preauth_not_reqd_mmorgan.png)

ASREPRoasting es similar a Kerberoasting, pero implica atacar el AS-REP en lugar del TGS-REP. No se requiere un SPN. Esta configuración se puede enumerar con PowerView o herramientas integradas como el módulo PowerShell AD.

El ataque en sí se puede realizar con el kit de herramientas [Rubeus](https://github.com/GhostPack/Rubeus) y otras herramientas para obtener el ticket para la cuenta objetivo. Si un atacante tiene permisos `GenericWrite` o `GenericAll` sobre una cuenta, puede habilitar este atributo y obtener el ticket AS-REP para crackear fuera de línea para recuperar la cuenta. s contraseña antes de desactivar el atributo nuevamente. Al igual que Kerberoasting, el éxito de este ataque depende de que la cuenta tenga una contraseña relativamente débil.

## Abuso de objetos de política de grupo (GPO)

La Política de grupo proporciona a los administradores muchas configuraciones avanzadas que se pueden aplicar tanto al usuario como a los objetos de la computadora en un entorno AD. La política de grupo, cuando se usa correctamente, es una excelente herramienta para reforzar un entorno de AD mediante la configuración de los ajustes del usuario, los sistemas operativos y las aplicaciones. Dicho esto, los atacantes también pueden abusar de la Política de grupo. Si podemos obtener derechos sobre un objeto de política de grupo a través de una mala configuración de ACL, podríamos aprovechar esto para el movimiento lateral, la escalada de privilegios e incluso el compromiso del dominio y como un mecanismo de persistencia dentro del dominio. Comprender cómo enumerar y atacar los GPO puede darnos una ventaja y, a veces, puede ser el boleto para lograr nuestro objetivo en un entorno bastante cerrado.

Se puede abusar de las configuraciones incorrectas de GPO para realizar los siguientes ataques:

- Agregar derechos adicionales a un usuario (como SeDebugPrivilege, SeTakeOwnershipPrivilege o SeImpersonatePrivilege)
- Agregar un usuario administrador local a uno o más hosts
- Crear una tarea programada inmediata para realizar cualquier cantidad de acciones

Podemos enumerar información de GPO utilizando muchas de las herramientas que hemos estado usando a lo largo de este módulo, como PowerView y BloodHound. También podemos usar [group3r](https://github.com/Group3r/Group3r), [ADRecon](https://github.com/sense-of-security/ADRecon), [PingCastle](https://www.pingcastle.com/), entre otros, para auditar la seguridad de los GPO en un dominio.
